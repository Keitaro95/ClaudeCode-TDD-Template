# 株主総会支援システム 要件定義書

## 目的

本ドキュメントは、株主総会支援システムおよび事前準備サポートツールの開発に関する要件定義書です。

前期に使用された株主総会支援システムの課題を解決し、役員様が質問に対してより迅速かつ適切に回答できる環境を提供すること、および事前準備における想定問答集の作成を効率化することを目的としています。

### システム構成要素

本システムは以下の2つの主要コンポーネントで構成されます。

1. **株主総会支援ツール**
   - 株主総会当日に使用するリアルタイム回答支援システム
   - 役員画面とオペレータ画面で構成

2. **事前準備サポートツール**
   - 株主総会の事前準備として想定問答集を作成するツール
   - 面談の音声認識、文字起こし、想定問答生成機能を提供

### システム方式・構成

#### アーキテクチャ概要

```
[役員タブレット (複数)]  ←→  [フロントエンド (Streamlit)]
[オペレータPC]          ←→  [フロントエンド (Streamlit)]
                              ↓
                        [Backend (FastAPI)]
                              ↓
                    ┌─────────┴─────────┐
            [Azure Speech Service]  [Claude Sonnet 4.5 API]
                    │                     │
            [Azure Blob Storage]   [Azure Cosmos DB]
                                          ↓
                                  [Vector Store]
```

#### 技術スタック

| レイヤー | 技術 | 選定理由 |
|---------|------|---------|
| **フロントエンド** | Streamlit | Python統一開発、迅速なプロトタイプ、リアルタイム更新対応 |
| **バックエンド** | FastAPI | 非同期処理、高パフォーマンス、Python統一 |
| **データベース** | Azure Cosmos DB | スケーラブル、ベクトル検索対応、Azure統合 |
| **ベクトルDB** | Azure Cosmos DB (Vector Search機能) | 想定問答のセマンティック検索 |
| **音声認識** | Azure Speech Service | リアルタイム文字起こし、話者分離 |
| **LLMモデル** | Claude Sonnet 4.5 | 長文処理能力、200Kコンテキスト、Azure統合可能 |
| **音声ストレージ** | Azure Blob Storage | 大容量音声データの長期保存 |
| **認証** | Azure AD (Entra ID) | エンタープライズSSO、セキュリティ |
| **リアルタイム通信** | Server-Sent Events (SSE) | Streamlitとの相性、シンプルな実装 |
| **デプロイ** | Azure App Service | PaaS、自動スケーリング、CI/CD統合 |

### 用語定義

| 用語 | 定義 |
|------|------|
| 役員画面 | 株主総会当日に役員が質問と回答を確認するためのタブレット画面 |
| オペレータ画面 | 株主総会当日にオペレータが質問の文字起こし、編集、回答生成を操作するPC画面 |
| 回答レベル | 質問に対する回答の詳細度（要約・詳細の2段階） |
| チェックモード | オペレータが確認・編集した回答を役員画面に送信するモード |

| RAG | Retrieval-Augmented Generation（検索拡張生成）- ベクトルDBから関連情報を取得してLLMに渡す手法 |
| 面談 | 株主総会前に実施される、想定される質問についてのヒアリング会議 |

## 業務要件

### 現状の課題

#### 株主総会支援ツールの課題

1. **UI/UXの問題**
   - 役員画面とオペレータ画面が共有されており、役員にとって不要な情報が表示される
   - 文字サイズが小さく、役員が回答を確認しにくい

2. **パフォーマンスの問題**
   - 回答生成に時間がかかり、即応性が低い
   - 質問の重要度に応じた適切な回答粒度の調整ができない

#### 事前準備の課題

1. **手作業の負担**
   - 各面談内容の整理に手間がかかる
   - 想定問答集の作成に時間がかかる

2. **データの活用不足**
   - 過去の面談内容や想定問答が体系的に蓄積されていない
   - 株主総会支援ツールとの連携が不十分

### 解決の方向性

#### 株主総会支援ツール

1. **専用UI/UXの提供**
   - 役員画面とオペレータ画面を分離
   - 役員画面は見やすさと操作性を最優先

2. **即応性の向上**
   - ダイレクトモードによる直接回答配信
   - 複数回答レベルによる柔軟な対応

3. **回答品質の向上**
   - 最新LLMモデル（Claude Sonnet 4.5）の採用
   - ベクトルDBを活用したRAGによる精度向上

#### 事前準備サポートツール

1. **自動化による効率化**
   - リアルタイム音声認識による文字起こし自動化
   - AI による想定問答の自動生成

2. **データの蓄積と活用**
   - 面談内容のデータベース化
   - 想定問答をベクトルDBに登録し、本番でも活用

### 構築後のフロー

#### 株主総会支援ツール（当日）

1. **事前設定**
   - オペレータが回答レベルの文字数を設定
   - 回答モード（チェック/ダイレクト）を選択

2. **質問受付**
   - 株主が質問を発言
   - Azure Speech Serviceがリアルタイムで文字起こし
   - オペレータが必要に応じて質問を編集・メモを追加

3. **回答生成**
   - 回答生成ボタンをクリック（目標レスポンス: 15秒以内）
   - バックエンドがベクトルDBから関連想定問答を検索
   - Claude Sonnet 4.5が要約・詳細の2レベルの回答を生成

4. **回答配信**
   - **チェックモード**: オペレータが回答を確認・編集後、送信
   - **ダイレクトモード**: 生成された回答を即座に全役員画面に配信

5. **役員による回答**
   - 役員が要約/詳細タブを切り替えて回答を確認
   - 必要に応じて文字サイズを調整
   - 回答内容を参考に口頭で回答

#### 事前準備サポートツール（事前準備期間）

1. **面談の実施**
   - 新規面談を作成
   - 面談中の音声をリアルタイムで文字起こし
   - オペレータが手動でメモを追加可能

2. **文字起こしの補正**
   - AIによる文字起こしの自動補正
   - 担当者が専門用語などを手動で修正
   - 修正済みテキストのインポート機能

3. **想定問答の生成**
   - 文字起こしから想定質問をAIが生成
   - 想定質問に対する回答をAIが生成
   - 担当者が内容を確認・編集

4. **想定問答のエクスポートと登録**
   - 完成した想定問答集をエクスポート
   - ベクトルDBに自動登録され、本番で活用可能

#### 1.3 バックエンド機能

##### 1.3.1 生成AI回答生成モジュール

| 項目 | 仕様 |
|------|------|
| LLMモデル | Claude Sonnet 4.5（200Kコンテキストウィンドウ） |
| 生成方式 | RAG（Retrieval-Augmented Generation） |
| プロンプト構成 | システムプロンプト + 関連想定問答 + 質問文 |
| 出力形式 | JSON（要約・詳細の2つのフィールド） |

**処理フロー:**

```
1. 質問文を受信
2. 質問文をベクトル化（Embedding API使用）
3. ベクトルDBで類似想定問答を検索（上位5件）
4. Rerankモデルで関連度を評価し、上位3件に絞り込み
5. プロンプトを構築:
   - システムプロンプト: 株主総会の回答生成の指示
   - 関連想定問答: 検索結果の想定問答
   - 質問文: ユーザー入力の質問
   - 文字数指定: 要約・詳細それぞれの文字数
6. Claude Sonnet 4.5 APIに送信
7. 生成された回答（要約・詳細）を返却
```

##### 1.3.2 ベクトルDB検索モジュール

| 項目 | 仕様 |
|------|------|
| データベース | Azure Cosmos DB（Vector Search機能） |
| Embeddingモデル | text-embedding-3-large（OpenAI） |
| ベクトル次元数 | 3072次元 |
| 類似度計算 | コサイン類似度 |
| 検索件数 | 初期検索: 5件、Rerank後: 3件 |

##### 1.3.3 Rerankモジュール

| 項目 | 仕様 |
|------|------|
| モデル | Cohere Rerank（または社内実装） |
| 入力 | 質問文 + 検索結果5件 |
| 出力 | スコア順に並び替えた上位3件 |
| 目的 | ベクトル検索の精度向上 |

##### 1.3.4 ベクトルDB更新モジュール

| 項目 | 仕様 |
|------|------|
| 更新タイミング | 事前準備サポートツールで想定問答をエクスポート時 |
| 処理内容 | 想定問答をEmbedding化してベクトルDBに登録 |
| バッチ処理 | 深夜帯に既存データの再Embedding（モデル更新時） |

##### 1.3.5 リアルタイム通信モジュール

| 項目 | 仕様 |
|------|------|
| プロトコル | Server-Sent Events (SSE) |
| エンドポイント | `/api/stream/officer/{session_id}` |
| メッセージ形式 | JSON（質問文、要約回答、詳細回答、タイムスタンプ） |
| ハートビート | 30秒ごとにping送信 |
| 接続管理 | セッションIDで複数役員画面を識別 |

**RAG処理フロー:**

```
1. 想定質問を受信
2. ベクトルDBで類似想定問答を検索（過去のデータ）
3. 検索結果を評価（Rerank）
4. プロンプトを構築:
   - システムプロンプト: 株主総会の想定問答生成の指示
   - 面談の文字起こし: 今回の面談内容
   - 関連想定問答: 過去の類似問答
   - 質問: 想定質問
5. Claude Sonnet 4.5 APIに送信
6. 生成された回答を返却
```

**注記:** Web検索機能は今回のスコープ外とする
