## テストケースを表にする時のポイント

まずテストには、この2つがあるらしい

1. userからの数値のinputに関するテスト - input入力などユーザーが入力して送信処理する。

2. システムの都合で条件判定・判断が加わるものに関するテスト - システムであらかじめ決めてる条件判定。会員の年齢とか選択プランで値段が変わる、等。

|--------|-------|
| 有効な管理番号（1000〜9999） | 1000, 5000, 9999 |
| 無効な管理番号（範囲外） | 999, 10000 |

異常系テストとは。異常がやってくる方向から分類します。

- 入力値の異常: ユーザーやクライアントから来ます。空文字、境界値超過、不正な形式など
- 状態の異常: ビジネスロジック。システム内部のデータに起因します。リソースが見つからない、すでに処理済み、権限不足など
- 環境の異常: 本番環境で起こりやすい。外部依存に起因します。ネットワーク障害、DB接続失敗、ディスク容量不足など
- 競合の異常: 並行処理に起因します。同時更新、デッドロック、レースコンディションなど

入力値の異常は前半の記事で書いたので省略

**空値の扱い**
境界値の中でも、特に扱いが難しいのが「空」という概念です。空値の扱いは設計上の判断が必要になります。

スペースのみ " "	トリムするか、エラーにするか
NULL	必須項目か、オプショナルか

テストを書くことで、こうした設計の曖昧さが明確になることがあります

### 状態の異常

#### 存在しないリソース

存在しないIDでリソースにアクセスしたとき、ちゃんと適切なエラー（404 Not Foundなど）が返ってくるかを確認します。
ユーザーが適当なIDを入力したり、
古いリンクを踏んだりしたときに
システムが「そんなの無いよ」と正しく伝える。

#### 削除済みリソース

削除されたDBレコードに再度アクセスしたときの動作を確認します。

ありがちなシナリオを考えてみましょう。
ユーザーが以前取得したレコードのIDを使って、再度そのレコードを取得しようとした。しかし、そのレコードは管理者によって削除されていた。よくある話です。

ここで問題になるのは、「404 Not Found」や「Record Not Found」で終わりにするのか、「このレコードは削除されました」と丁寧に伝えるのか、という設計上の判断です。

これは論理削除と物理削除で挙動が異なります。論理削除（deleted_atフラグを立てるだけ）なら、レコード自体は残っているので「削除済み」というステータスを返せます。物理削除（DELETEでレコードごと消す）なら、そもそもレコードが存在しないため、「レコードが見つかりません」を返すしかありません。

どちらの設計を採用しているかで、テストの期待値も変わってきます。まずは自分のシステムがどちらを採用しているか確認しましょう。

#### 処理中のリソース

処理中（アップロード中、変換中など）のリソースにアクセスしたときの動作を確認します。動画のエンコード中、大きなファイルのアップロード中、バッチ処理の実行中...。こういった「まだ準備ができていない」状態をクライアントに適切に伝えられるかがポイントです。

HTTP 202 Accepted を返すのか、専用のステータスを返すのか。進捗を問い合わせるエンドポイントがあるのか。このあたりの設計が適切に実装されているかをテストします。

#### 不正な状態遷移

状態遷移が定義されているシステムでは、不正な遷移を試みたときの動作を確認します。例えば、注文のステータス遷移を考えてみましょう。

```
# 正常な遷移
作成 → 確定 → 発送 → 完了
```

では、こんな不正な遷移はどうなるでしょうか？

```
# 不正な遷移の例
作成 → 完了（確定と発送をスキップ）
完了 → 作成（逆方向の遷移）
```

こういった「ありえない遷移」をAPIから叩いたときに、システムがちゃんと拒否してくれるか。これは意外と見落としがちなテスト項目です。



### 環境の異常

環境の異常は、テストが最も難しい領域です。開発環境では再現しにくいですが、プロダクション環境では必ず発生します。

ローカルで動いたからといって、本番で動く保証はありません。開発環境は、ある意味で嘘をつきます。ネットワークは常に安定し、データベースは常に応答し、ディスクは無限にあります。そんな理想的な環境でテストしても、現実の障害には備えられません。

だからこそ、どういう異常が起こりうるかを知っておくことが重要です。

まずは以下のような基本的な異常パターンを理解しておきましょう。

#### ネットワーク障害

- 通信経路の遮断
- タイムアウト
- オンライン→オフラインの遷移

対応方法としては、タイムアウト設定、リトライ、サーキットブレーカーなどがあります。

**サーキットブレーカー**とは、外部サービスへのリクエストが連続して失敗したとき、一時的にリクエストを遮断する仕組みです。電気のブレーカーが過電流を検知して回路を遮断するのと同じ発想で、障害の連鎖を防ぎます。

#### データベース障害

- DB応答不可
- コネクションプール枯渇
- デッドロック

対応方法としては、コネクションプールの適切な設定、リトライ、タイムアウトなどがあります。

#### 外部サービス障害

- API応答不可
- レートリミット
- 予期しないレスポンス形式

対応方法としては、サーキットブレーカー、フォールバック、キャッシュなどがあります。

#### リソース枯渇

- ディスク容量不足
- メモリ不足
- ファイルディスクリプタ枯渇

リソース枯渇は、テストで再現するより監視とアラートで早期に検知する方が現実的です。とはいえ、リソースが枯渇したときにシステムがどう振る舞うか（gracefulに停止するか、エラーメッセージを出すか）は、設計段階で決めておく必要があります。

#### カオスエンジニアリングの実践
Netflixが提唱したこのアプローチでは、本番環境に意図的に障害を注入し、システムの回復力を検証します


#### レジリエンスパターン

環境の異常に備えるには、コードにレジリエンスパターンを組み込む必要があります。先に紹介したサーキットブレーカーに加え、以下のパターンが重要です。

- **Bulkhead（バルクヘッド）**: 船の隔壁のように、リソースを区画化して一部の障害が全体に波及することを防ぎます
- **Retry with Exponential Backoff（指数バックオフ付きリトライ）**: 失敗したら1秒後、2秒後、4秒後…と間隔を広げてリトライします。リトライストームを防止しながら一時的障害から回復します

これらのパターンを実装したら、カオスエンジニアリングで実際に障害を注入し、期待通りに動作するか検証します。パターンを実装しただけでは不十分で、テストして初めて信頼できます。

### 競合の異常

本番環境で、複数のユーザーやプロセスが同時にリソースにアクセスすると、競合が発生しえます。
「競合が起きたらどうなるか」を事前に設計します。

#### 同時更新（DBレコードの競合）

典型的なシナリオを考えてみましょう。

1. ユーザーAがデータを取得
2. ユーザーBが同じデータを取得
3. ユーザーAが更新を実行
4. ユーザーBが更新を実行 → **どうなるべきか？**

ユーザーBの更新時点で、データはすでにユーザーAによって変更されています。このとき、システムはどう振る舞うべきでしょうか。

**どうすべきか：3つの対応方法**

| 方法 | 仕組み | 向いているケース |
|------|--------|------------------|
| **楽観的ロック** | データ取得時にバージョン番号を記録し、更新時に照合。バージョンが変わっていればエラーを返す | 在庫数など「先の更新が失われると困る」データ |
| **悲観的ロック** | 更新する意思を示した時点で排他ロックを取得。他者は同じデータを更新できなくなる | 金融取引など確実性が最優先のケース |
| **最後の更新が勝つ** | 競合を検出せず、後から来た更新で上書き | ユーザープロフィールなど「最新の状態が正」でよいデータ |



#### ボタン連打（UIの競合）

「送信ボタンを押したけど反応がない。もう一度押そう」。

ネットワークが遅いとき、人は本能的に連打します。

「購入する」ボタンを連打したら2回購入されてしまった、という事故は避けたいところです。

**どうすべきか：フロントエンドとバックエンドの両方で対策**

| レイヤー | 対策 | 説明 |
|----------|------|------|
| **フロントエンド** | ボタン無効化 | 送信中はボタンをdisabledにして連打を物理的に防ぐ |
| **フロントエンド** | デバウンス | 一定時間内の連続クリックを1回とみなす |
| **バックエンド** | 冪等性の担保 | リクエストIDを使い、同一リクエストを検出して重複処理を防ぐ |

フロントエンドだけでは不十分です。ネットワークの遅延でリクエストが重複送信されることがあります。バックエンド側でも冪等性を担保し、「同じリクエストが2回来ても、処理は1回だけ」という設計にしておきましょう。

## ステータスコードの使い分け

ステータスコードとは、サーバーがクライアント（ブラウザやアプリ）に返す3桁の数字です。この数字を見れば、リクエストが成功したのか、失敗したのか、何が原因なのかが分かります。

### HTTP ステータスコード一覧

| コード | 名称 | 説明 | 使用場面 |
|--------|------|------|----------|
| 400 | Bad Request | 入力値が不正 | バリデーションエラー、必須項目の欠落、形式不正 |
| 401 | Unauthorized | 認証失敗 | ログインが必要、トークン切れ、認証情報なし |
| 403 | Forbidden | 権限不足 | ログイン済みだがアクセス権がない |
| 404 | Not Found | リソースが存在しない | 指定IDのレコードがない、URLが間違っている |
| 409 | Conflict | 競合 | 同時更新、重複登録、楽観的ロック失敗 |
| 429 | Too Many Requests | レートリミット | リクエストが多すぎる、API制限超過 |
| 500 | Internal Server Error | サーバー内部エラー | 予期しない例外、バグ |
| 503 | Service Unavailable | サービス利用不可 | メンテナンス中、依存サービス障害 |

### gRPC ステータスコード一覧

gRPCはGoogleが開発した高速な通信方式です。HTTPとは異なる独自のステータスコードを持ちます。

| コード | 説明 | HTTPとの対応 |
|--------|------|--------------|
| INVALID_ARGUMENT | 入力値が不正 | 400 Bad Request |
| NOT_FOUND | リソースが存在しない | 404 Not Found |
| ALREADY_EXISTS | リソースが既に存在 | 409 Conflict |
| FAILED_PRECONDITION | 前提条件不成立 | 400 / 409 |
| PERMISSION_DENIED | 権限不足 | 403 Forbidden |
| RESOURCE_EXHAUSTED | リソース枯渇（レートリミット等） | 429 Too Many Requests |
| UNAUTHENTICATED | 認証失敗 | 401 Unauthorized |
| INTERNAL | サーバー内部エラー | 500 Internal Server Error |
| UNAVAILABLE | サービス利用不可 | 503 Service Unavailable |

## セキュリティ観点でのエラー設計

### 問題：正しいエラーコードがセキュリティホールになる

他ユーザーのリソースへのアクセスには、エラーコードの選び方に注意が必要です。

**素直な設計（問題あり）**

```
リクエスト: GET /users/123/private-data
ケース1: リソースが存在し、権限がない → 403 Forbidden
ケース2: リソースが存在しない → 404 Not Found
```

この設計では、攻撃者がIDを総当たりで試したとき、403が返れば「このIDのリソースは存在する」と分かってしまいます。

**つまり、正しいエラーコードを返すことが、セキュリティホールになるという逆説です。**

### 対策：存在確認を隠蔽する

**推奨設計**

```
リクエスト: GET /users/123/private-data
ケース1: リソースが存在し、権限がない → 404 Not Found
ケース2: リソースが存在しない → 404 Not Found
```

「存在するが権限がない」と「存在しない」を区別できないようにすることで、攻撃者に情報を与えません。

### 設計指針

| シナリオ | 推奨レスポンス | 理由 |
|----------|---------------|------|
| 他ユーザーの非公開リソースにアクセス | 404 Not Found | 存在確認を防ぐ |
| 自分のリソースで権限不足 | 403 Forbidden | 自分のリソースなので存在は自明 |
| 認証されていない状態でアクセス | 401 Unauthorized | ログイン誘導のため |
| 公開リソースで権限不足 | 403 Forbidden | 公開情報なので隠す意味がない |

### 実装例

```python
def get_resource(resource_id: str, current_user: User):
    resource = db.get(resource_id)

    # リソースが存在しない
    if resource is None:
        raise NotFoundError()

    # 他ユーザーのリソース → 404を返す（存在を隠す）
    if resource.owner_id != current_user.id:
        raise NotFoundError()  # 403ではなく404

    return resource
```

### 採用事例

- **GitHub**: プライベートリポジトリに権限なしでアクセスすると「404 - This is not the web page you are looking for」と表示
- **Slack**: 他ワークスペースのチャンネルにアクセスしても存在を確認できない

### 注意点

これは「嘘をつく」のではなく、「必要以上の情報を与えない」という設計です。

ログやモニタリングでは、実際の原因（権限不足なのか、存在しないのか）を記録しておくことで、デバッグや監査には影響しないようにします。

```python
# ログには真の原因を記録
logger.warning(f"Access denied: user={current_user.id}, resource={resource_id}, reason=permission_denied")
# クライアントには404を返す
raise NotFoundError()
```
